## 7. Problemes identifies

### P0 — Performance GPU

- **14.7 sec** par extraction LLM = bottleneck majeur
- GTX 1650 (4GB VRAM) saturee a 89% (3666/4096 MB)
- **Resolution attendue**: RTX 3060 (12GB) → modele 14B, temps ~3-5 sec

### P1 — Taux d'echec LLM

- **13.4%** des tentatives LLM echouent (41/307)
- Cause principale: JSON tronque par Phi-4 Mini (model trop petit)
- **Resolution attendue**: 14B avec meilleure adherence au schema JSON

### P2 — Early-stop excessif

- **580 events** early-stop / 307 tentatives = 1.9x overhead
- Le modele genere des `}` prematures avant d'avoir tous les champs
- **Action**: Optimiser le prompt d'extraction pour le prochain modele

### P3 — Doublons de threads

- 5 paires de doublons detectees et fusionnees manuellement
- Thread matching ONNX @ 0.60 n'a pas fusionne des contenus quasi-identiques
- **Action**: Evaluer le threshold ou la qualite des embeddings sur du code source

### P4 — Extraction degeneree

- 3 threads (10%) avec titre = instruction du prompt ("max 50 chars, descriptive")
- Le modele a regurgite l'instruction au lieu d'extraire
- **Resolution attendue**: 14B + prompt optimise
